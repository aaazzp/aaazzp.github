Android之Activity的生命周期

`

//按加载顺序
@Override
public void onCreate(){

/* 
在这个方法中不适宜进行太耗时的操作，避免造成加载页面卡顿
一般只在这个方法中加载view即可。
该方法为瞬时的，很快就会进行onStart()
*/

}

@Override
public void onStart(){

/* 
进行到该方法时，界面为可见但不可交互。
暂时不清楚在该方法下，应该进行那些操作
个人理解为可以进行一些配置操作。
该方法为瞬时的，很快就会进行onResume()

官方建议：onStop里面做了哪些清除的操作，就该在onStart里面重新把那些清除掉的资源重新创建出来
*/

}

@Override
public void onResume(){

/* 
进行到该方法时：页面为可见且可交互
应该实现onResume()来初始化那些在onPause方法里面释放掉的组件，
并执行那些activity每次进入Resumed state都需要的初始化动作 (例如开始动画与初始化那些只有在获取用户焦点时才需要的组件)

且在onPause()后重新进入该activity时，会重新运行一次该方法
所以可在该方法下，进行数据的加载，以便activty onPause()之后，用户
重新进入当前activy导致无数据显示。
*/

}

@Override
public void onPause(){

/* 
当activity仍有部分可见时，就会运行到这个方法，如某一个半透明的activity以对话框的
形式弹出时。
在这个方法中应该释放一些资源，如暂停动画，暂停视频的播放之类的。
若，是启动下一个activity时，在完成这个方法后，便会执行
下一个activity的onCreate()方法，所以在这里不应该进行太耗时的操作
避免造成切换过慢。
*/

}

@Override
public void onStop(){

/* 
当整个activity不可见时，便会调用该方法
在这里应该释放那些不再需要的所有资源。
系统会在需要内存空间时摧毁该activity的实例(和栈结构有关，通常back操作会导致前一个activity被销毁)
在这里可以进行一些用户确定需要储存的数据操作。
官方建议：
onStop里面做了哪些清除的操作，就该在onStart里面重新把那些清除掉的资源重新创建出来
*/

}

@Override
public void onRestart(){

/* 
当运行至onStop()后，用户重新进入当前activity时，便会调用该方法。
*/

}

@Override
public void onDestroy(){

/* 
当系统销毁该activity时便会执行当前方法
在这里需要去清除那些可能导致内存泄漏的地方
需要确保那些线程都被destroyed并且所有的操作都被停止
*/

}

/*
注意：当该activity因为系统资源紧张而导致被销毁的时候，
系统会在用户回到这个Activity时有这个Activity存在过的记录，
系统会使用那些保存的记录数据（描述了当Activity被Destory时的状态）来重新创建一个新的Activity实例

一般调用onSaveInstanceState()保存在Bundle中（为了使Android系统能够恢复Activity中的View的状态，每个View都必须有一个唯一ID）。

然后调用onRestoreInstanceState()回复数据。

*/`
